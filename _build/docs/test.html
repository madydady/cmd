Introduction¶

Review of the TMS/X functionality¶

The TMS/X application is used in order to prepare and send cash operations to the RTGS and ACH for processing.

The TMS/X also allows users to monitor:

  • processing of transactions and requests, have been sent to the Central node,
  • reports on transactions and message exchange;
  • other information, related to the bank operations.

Workstation requirements and settings¶

Workstation requirements¶

Before starting to work with the Web interface system, make sure that your computer has been installed Microsoft Internet Explorer 10.0 or higher. The Java script should be enabled in web-browser settings. A client PC should have at least 512 MB of RAM and at least 1 GB free space on a hard drive.

Browser proxy-server settings¶

If you operate via a proxy-server, find out its address and the port number from your system administrator. Set the received information in the correspondent fields of the IE “Tools” - “Internet Options” window (“Connection” – “LAN Settings” – “Proxy server”). If the proxy server will not be used, then no additional settings are required.

ActiveX component installing and setting¶

No special installation procedures are required at the participant’s site, but when a user connects to the system the first time, the user should allow installing ActiveX component (in case user is going to sign and send transactions via TMS/X GUI manually). Web interface workplace is built on thin technologies and uses the standard browser for viewing its GUI.

To run genuine Windows validation, you have to install an ActiveX control. The Internet Explorer might not be configured to download ActiveX controls for security reasons, or you might have declined to install the ActiveX control when the Internet Explorer prompted you to download, install, or run the control. Do the following actions:

  • Open the Internet Explorer;
  • Click the Tools menu, and then click the Internet Options;
  • On the ‘Security’ tab (trusted sites), click the Custom level button;
  • Scroll down the Security Settings list until you see the ‘ActiveX controls and plug-ins’;
  • For Automatic prompting for the ActiveX controls, click Enable;
  • Scroll down to the ‘Download signed ActiveX controls’ and click Enable or Prompt;
  • Scroll down to the ‘Run ActiveX controls and plug-ins’ and click the Enable or Prompt;
  • Scroll down to the ‘Script ActiveX controls, marked safe for scripting’ and click the Enable or Prompt;
  • Click the OK button and then click the OK again. (You may need to restart your computer.).

Adding to Trusted Sites¶

Add the URL https://<server>:<port>/portal/ to trusted sites, where:

  • <port> - TCP port for SSL connection to the TMS/X;
  • <server> - name or ip address of the TMS/X server.

To do this, follow the steps:

  • In the Internet Explorer, click Tools > Internet Options, and then click the ‘Security’ tab;
  • In the ’Select a Web content’ zone s box, click Trusted Sites > Sites to specify its current security setting;
  • In the ‘Add this Web site to the zone’ box, type the URL of the TMS/X site, and then click the Add button

Click the OK button two times to accept the changes and return to the Internet Explorer.

Using only TLS 1.1 and TLS 1.2¶

In case you have to only use the TLS 1.1 and TLS 1.2, make the following setting:

  • Left-click the gear icon;
  • Select “Internet options” from the dropdown menu;
  • Click the “Advanced” tab, scroll down and deselect “SSL 2.0” “SSL 3.0” and “TLS 1.0”;
  • Select “TLS 1.1” and “TLS 1.2”;
  • Click the OK button to accept your changes (it should take effect immediately, but you may need to refresh your browser).

Disable compatibility view¶

In case you are using the IE 11version:

  • Press the ‘Alt’ key on your keyboard, this will make a menu bar appear;
  • Click on the Tools menu tab;
  • Select the Compatibility View settings option;
  • Uncheck the “Display intranet sites in Compatibility View” option.

User Role Hierarchy and Allocation¶

The TMS/X Workplace supports the following allocation of the user roles (each role defines set of access rights (permissions) to monitoring data and making operations):

  • System Administrator – gives the permission to dictionaries, authorization profiles and switchover management;
  • Security administrator – gives the permission to create user access and authorities’ management tasks;
  • Security officer – gives the permission to approve user access and authorities’ management tasks;
  • Operator – gives the permission create, edit and make transactions and transaction requests;
  • Controller – gives the permission to authorize transaction and transaction requests.

The TMS/X administrator may assign one or several roles from the list to any user.

Each of the TMS/X users connects to the TMS/X by using his own login and password. After the user is connected, the TMS/X identifies the user’s role(s), which are definitely associated with the user credentials in the system. The contents of the TMS/X pages and menus will correspond to the identified user role.

Processing transactions¶

Authorization: basic principles¶

Authorization in the TMS/X can be applied to a particular type of documents. The authorization itself is a procedure that assumes specified actions from user(s), other than the user, who created the document. Dependently on settings, which are provided by the TMS/X Administrator (described in the separate TMS/X Administrator Guide, supplied by the CMA), the authorization procedure may be either applied or not to the particular documents (or their types) and its rules may flexibly be changed.

All the authorization actions are made by approvers. An approver in the TMS/X is a person (user) who has an authority to authenticate and/or change the document. To be an approver, such user(s) must own at least one of the corresponded roles that is also managed by the TMS/X Administrator.

Making authorization actions¶

After the document is issued by user-maker and saved in the TMS/X local database, the TMS/X tries to pick up an authorization profile, which would be applied to the document (based on appropriate settings, have been done by the Administrator). In the case the profile is found, local status of the document changes from “Created” to “Waiting for Authorization”, and users-approvers can make the following authorization actions:

  • Approve the document. The Approver checks the document attributes via
    popping up the view form and calls the ‘Approve’ command, provided (s)he didn’t notice any kind of mistakes. The document will move to the next processing step (is ready to be sent);
  • Decline the document. In the case any errors were found, the Approver
    user calls the ‘Decline’ command. The document will move back to the previous step for repairing.

The both actions are logged in the document processing audit.

Return transaction¶

Login the TMS/X as the user, who has authority to create return transactions. This option is available for payments in ‘Received’ Local status and settled by Central node.

Open the Transactions monitoring table, choose the necessary document in there and click its Reference hyperlink. Press button Return batch at the bottom of the opened form:

In the opened window choose the instructions to return, enter or choose return reason for each instruction and press button Create return. The new payment form will be generated and opened automatically in ‘Waiting for authorization’ status.

Message audit¶

Call the Monitoring > Message audit item of the TMS/X main menu to open a table of messages, which was sent both from the TMS/X to the RTGS or ACH and from the RTGS or ACH to the TMS/X, and from/to CBS:

The table has a set of the following fields:

  • ID – system numeric identifier of the message (a hyperlink)
  • Reference – reference number of the message;
  • Content type – code of the message type according to the content
    (MX as default);
  • Type – code of the message type according to the message format;
  • Direction – code of the message flow direction (from/to RTGS/ACH
    or CBS);
  • State – code of the result of the message processing;
  • State Description – brief textual description of the result of
    the message processing;
  • Creation date – date and time, when the message was created.

Use the table filtering option to find quickly a particular message (see the section 2.4.3). Select the message in the table and click the ID hyperlink to get access to the message details and related objects. The message details form will be opened in a separate window (the message type, direction and reference will be put into the header of the window):

Depending of the message type the form could have following tabs:

  • The Original tab displays main attributes of the selected message (ID, state, date) and content of the message body in original format;
  • The Links tab displays list of the TMS/X objects, which are linked to the selected message (such as, another message, transaction, event, etc.). Click the Type hyperlink in the list to view details of the selected linked object in a separate window.
Saving message in a file¶

In order to export message data into a file, select the message in the table, open its details form and click the (download) icon in the right upper corner.

The standard file saving message box will be opened. Choose the ‘Save’ or the ‘Save As’ option (in the case of ‘Save As’, define the file destination and its name), and then press the Save in the pop-up standard file browser window. The file with the message content will be saved in the target folder.

Printing a message¶

In order to print message data, select the message in the table, open its details form and click the (print) icon in the right upper corner.

The standard dialog box for printing option will be opened. Choose the printer setting and press the Print button. The printing page will be sent to the specified printer device.

Reconciliation¶

Call the Monitoring > Reconciliation item of the TMS/X main menu to open to view result of reconciliation based on statements (MT950) received from (RTGS and ACH):

The Reconciliation menu item contains to tables:

  • Reports – contains the list of reconciliation reports (MT950).
    The Reports table contains the following columns:
  • Reference – the reference number of the message;
  • Type – message type;
  • Name – name of the message type;
  • Status – status of the report processing. It’s ‘Closed’ – when correspondent payments from the report reconciled automatically (has been found in TMS/X database and linked with records in the report) or ‘Manual reconciliation required’ – when some of correspondent payments from the report not linked (hasn’t been found in TMS/X database);
  • Subsystem – origin of the report (RTGS or ACH);
  • Creation date – date of the message creation;
  • Transactions – contains the list of transactions of selected
    report. The Transactions table contains the following columns:
  • Reference – the reference number of the payment;
  • Value date – date of the payment creation;
  • Amount - amount of payment;
  • Currency – currency of payment;
  • Status – reconciliation status of each payment from the report. It’s ‘Automatically reconciled’ – when correspondent payment from the report reconciled automatically (has been found in TMS/X database and linked with records in the report) or ‘Manually reconciled’ – when some of correspondent payments where reconciled manually.

For manual reconciliation choose report in ‘Manual reconciliation required’ status in the Reports table, mark necessary payments in the Transactions table and click button Reconcile. The report status will change into ‘Closed’ and payments status into ‘Manually reconciled’.

Custom reports¶

Call the Reports item to of the main menu to get a list of the TMS/X custom reports:

The following reports are available in the system:

  • Roles audit trial - the report on roles profiles’ changes for the
    selected period. Contains the list of roles, actions and their dates and time;
  • Users audit trial – the report on user profiles’ changes for the
    selected period. Contains the list of users, actions and their dates and time;
  • User login status report – the report on users never logged in,
    users logged in within last selected days and users disabled since selected days;
  • Periodic review – review user info report on current date.
    Contains the list of users and assigned roles;
  • Last password change report – the report on users didn’t change
    their password for more than selected days before selected data;
  • Granted user authorities report – the report on users’ roles and
    authorities on current date;
  • Failed login report – the report on failed login users for the
    selected period. Contains the list of users, dates and time of failed login and reason description;
  • Local user activity report – the user activity report on selected
    date;
  • Entitlements report – the report on created roles;
  • Report on settled payments – the report on settled transactions
    on selected date;
  • Report on rejected or cancelled transactions – the report on
    transactions in rejected, cancelled or refused status on current date;
  • Internal payment report – the report on internal transactions in
    selected currency to on selected date;
  • Outgoing transactions report – the report on outgoing
    transactions on selected date;
  • Incoming transactions report – the report on incoming
    transactions on selected date.

To view the report choose necessary menu item, set parameters of the report, choose the report file format and click the button Print:

The report will be generated in selected file format. To open or save the report click the correspondent button in the standard browser pop up dialog box. System supports following files formats of the reports – pdf, xls, rtf, csv, html, docx.
